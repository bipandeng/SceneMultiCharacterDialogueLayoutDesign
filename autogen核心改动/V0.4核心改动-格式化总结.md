## 1️⃣ 分层架构：Core + AgentChat 双层

**v0.2 痛点**：单体 `ConversableAgent` 啥都干，底层通信机制与业务逻辑高度耦合，改不动。

**v0.4 设计**：

- **Core API**：基础层，提供**事件驱动的 Actor 框架**（`RoutedAgent`），用于创建自定义代理工作流。
    - 适用人群：高级架构师、科研人员。
    - 比喻：餐厅**后厨定制区**——只提供员工和"传纸条"机制。
- **AgentChat API**：基于 Core 构建的**任务驱动高级框架**，开箱即用。
    - 适用人群：普通开发者、业务团队。
    - 比喻：餐厅**前厅套餐区**——开箱即用的精装样板间。

> **关系**：AgentChat 是 AutoGen 的**替代品**（亲儿子顶替养子），微软在 Core 新地基上重写上层。

## 2️⃣ 客户端缓存：默认关闭 + Wrapper 模式

**v0.2 痛点**：缓存默认开启、绑死在 `llm_config` 字典里、不支持分布式。

**v0.4 机制**：

```python
# 1. 底层 OpenAI 客户端（不带缓存）
openai_client = OpenAIChatCompletionClient(model="gpt-4o", api_key="...")

# 2. 缓存存储器
cache_store = DiskCacheStore(path="./my_cache")  # 或 RedisStore(...)

# 3. 用缓存外壳把底层客户端包起来
cached_client = ChatCompletionCache(client=openai_client, store=cache_store)
```

**对比**：

| 维度 | v0.2 | v0.4 |
|------|------|------|
| 默认状态 | 默认开启 | **默认关闭**，需手动包 Wrapper |
| 配置方式 | `llm_config["cache_seed"]` | `ChatCompletionCache(client=..., store=...)` |
| 存储位置 | 默认本地磁盘 | 灵活：`DiskCacheStore`（本地）或 `RedisStore`（分布式） |
| 适用场景 | 单机原型 | 企业级多节点 |

**价值**：省钱（按 Token 计费）、省时（毫秒级命中）、一致性（相同输入稳定输出便于 Debug）。

## 3️⃣ 自定义 Agent：从回调注册到 OOP 继承

**v0.2 老办法（贴便利贴）**：

```python
def reply_func(recipient, messages, sender, config):
    return True, "自定义回复"

conversable_agent.register_reply([ConversableAgent], reply_func, position=0)
```

- 😵 痛点：参数靠猜（recipient/sender/messages 容易晕）；`position` 控制优先级是黑盒。

**v0.4 新办法（定制岗位）**：

```python
class CustomAgent(BaseChatAgent):
    async def on_messages(self, messages, cancellation_token):
        return Response(chat_message=TextMessage(content="自定义回复"))

    async def on_reset(self, cancellation_token):
        pass

    @property
    def produced_message_types(self):
        return (TextMessage,)
```

- 😊 优势：参数清晰（`messages` 就是收到的消息）、告别黑盒、职责单一。

**对比**：

| 维度 | v0.2（贴便利贴） | v0.4（OOP 继承） |
|------|-----------------|------------------|
| 设计模式 | 面向过程 / 打补丁 | 面向对象 |
| 核心动作 | `register_reply()` | 重写 `on_messages()` |
| 参数理解 | 靠猜 | 清晰 |
| 多逻辑冲突 | `position` 控序（黑盒） | 方法内自控 |

## 4️⃣ 状态管理：一键存档 / 读档

**v0.2 痛点**：无内置存档，需手动掏 `chat_messages` 属性，团队场景要挨个 Agent 处理。

**v0.4 新办法**：

```python
# 存档
state = await assistant.save_state()
with open("assistant_state.json", "w") as f:
    json.dump(state, f)

# 读档（时光倒流）
await assistant.load_state(state)
```

**团队状态**：对 `Team` 对象调用 `save_state()` / `load_state()`，**整个群里所有 Agent 的记忆 + 当前轮到谁 + 任务进度**整体打包。

**应用场景**：

- 断点续传：用户今天聊一半、明天接着聊。
- A/B 测试：岔路口存档，分别尝试方案 A / B。
- 容错机制：AI 走偏后回退到正确节点。

**对比**：

| 维度 | v0.2 | v0.4 |
|------|------|------|
| 存档方式 | 手动提取 `chat_messages` | `save_state()` |
| 读档方式 | 初始化时手动塞回 | `load_state()` |
| 团队存档 | 遍历每个 Agent | 对 Team 一次调用，集体存档 |
| 数据格式 | 杂乱的字典列表 | 标准化 JSON |

## 5️⃣ State vs 业务库：两套不同的"账本"

**核心结论**：iOS App 中有两套系统同时工作，互不替代。

| 维度 | 业务数据库（MySQL/Mongo） | AutoGen State（save_state） |
|------|--------------------------|----------------------------|
| 大白话 | 法院庭审笔录 | 法官脑子里的记忆 |
| 主要用途 | 给 iOS App 展示 UI 聊天气泡 | 给 AI 恢复上下文记忆 |
| 数据位置 | App 服务器（主）+ 手机本地（缓存） | 只在 App 服务器 |
| 包含内容 | 发送者、时间、文本 | 系统提示词、工具调用状态、内部变量 |
| 给谁看 | 用户（人类） | AutoGen 框架（机器） |

**用户关 App → 重开 → 接着聊** 的完整流程：

1. 打开 App：iOS 从业务库拉取历史消息 → 显示在屏幕（**未用 AutoGen State**）。
2. 继续聊天：iOS 发消息 → 后端先 `load_state()` 恢复 AI 记忆 → AI 回复 → 后端调 `save_state()` 再存回。
3. 关键：**`save_state` 导出的 JSON 不能直接给 iOS 渲染 UI**，格式不匹配。

**开发建议**：

- 不要用 `save_state` 替代聊天记录库（仍需建 `messages` 表）。
- 后端为每个 Session 维护一个 Agent 实例，发消息时先 `load_state`、再 `save_state`。
- iOS 端不感知 AutoGen 存在。

## 6️⃣ 代码执行：程序员 + 测试员解耦

**场景**：让 AI 自己写代码 + 自己运行 + 改 Bug。

```python
# 程序员（动脑）
assistant = AssistantAgent(
    name="assistant",
    system_message="所有代码必须用 python 写。任务完成只能回复 'TERMINATE'。",
    model_client=model_client,
)

# 测试机（动手）
code_executor = CodeExecutorAgent(
    name="code_executor",
    code_executor=LocalCommandLineCodeExecutor(work_dir="coding"),
)

# 轮流干活群
group_chat = RoundRobinGroupChat(
    [assistant, code_executor],
    termination_condition=TextMentionTermination("TERMINATE") | MaxMessageTermination(10),
)

# 实时直播
stream = group_chat.run_stream(task="Write a python script to print 'Hello, world!'")
await Console(stream)
```

**v0.4 解耦哲学**：写代码只管写（`Assistant`），跑代码只管跑（`Executor`）。要换 Docker / 远程执行？只换 `CodeExecutor` 即可，程序员代码零改动。

## 7️⃣ 工具调用：独行侠模式

**v0.2 老办法（双人相声）**：必须雇两人——`tool_caller`（动脑）+ `tool_executor`（动手），还需 `initiate_chat` + `register_function` 复杂绑定。

**v0.4 新办法（一全包）**：

```python
def get_weather(city: str) -> str:
    return f"{city}的天气是72度，晴天。"

assistant = AssistantAgent(
    name="assistant",
    model_client=model_client,
    tools=[get_weather],              # 直接挂装备
    reflect_on_tool_use=True,         # 拿到结果后让 AI 润色
)

# 直接对一个 Agent 调用
response = await assistant.on_messages(
    [TextMessage(content="北京天气如何?", source="user")],
    CancellationToken(),
)
```

**流程**：

1. 用户问"北京天气" → AI 脑子一转发现挂了 `get_weather`。
2. **内部**调用 `get_weather("北京")`（不发出群聊消息）。
3. `reflect_on_tool_use=True` → 润色成"北京今天 72 度，晴朗，适合出门"。

**对比**：

| 维度 | v0.2 | v0.4 |
|------|------|------|
| Agent 数量 | 必须 2 个 | 1 个 |
| 工具绑定 | `register_function` 双向绑定 | 初始化传 `tools=[...]` |
| 执行逻辑 | 后台 `initiate_chat` | 内部自动闭环 |
| 异步支持 | 弱 | 原生 `async/await` |

## 8️⃣ 任务汇报：从保姆管家到冷酷记录仪

**v0.2 `ChatResult`（保姆管家）**：

```python
chat_result = tool_executor.initiate_chat(...)
chat_result.summary       # 自动调 LLM 写摘要
chat_result.chat_history  # 过滤后的纯对话
chat_result.cost          # 自动算美元
chat_result.human_input   # 单独提取人类输入
```

**v0.4 `TaskResult`（冷酷记录仪）**：

```python
task_result = await team.run(...)
task_result.messages  # 唯一字段：完整的消息流水（人话 + 工具调用 + 报错）
```

**v0.4 删了什么 / 为什么**：

| 字段 | v0.2 状态 | v0.4 状态 | 怎么自己搞定 |
|------|----------|----------|--------------|
| `summary` | ✅ 自动生成 | ❌ 删 | 拿 `messages` 自行调 LLM 总结 |
| `chat_history` | ✅ 纯对话 | 改名 `messages`（含工具调用） | 直接遍历 |
| `human_input` | ✅ 单独字段 | ❌ 删 | 在 `messages` 里找 `source == "user"` |
| `cost` | ✅ 美元 | ❌ 删（只给 Token 数） | 自行乘以模型单价 |

**哲学**：解耦——框架只负责忠实记录，至于总结/算钱/过滤是业务层的事。

## 9️⃣ 群聊：去中间商 + 实体化

**v0.2 老办法（线下会议）**：

- 三件套：`GroupChat` + `GroupChatManager` + `UserProxy`。
- 散会条件写在**员工个人合同**（`is_termination_msg`）里，逻辑分散。
- 😵 必须 `UserProxy.initiate_chat(...)` 才会开始。

**v0.4 新办法（线上协作群）**：

```python
writer = AssistantAgent(name="writer", ...)
critic = AssistantAgent(name="critic", ...)

termination = TextMentionTermination("APPROVE")

group_chat = RoundRobinGroupChat(
    [writer, critic],
    termination_condition=termination,
    max_turns=12,
)

# 老板直接对群派活
stream = group_chat.run_stream(task="写一个机器人发现感情的短篇故事。")
await Console(stream)
```

**两种群类型**：

- `RoundRobinGroupChat`：死板轮流（A → B → A → B）。
- `SelectorGroupChat`：智能调度（用 LLM 动态选下一个发言者）。

**对比**：

| 维度 | v0.2 | v0.4 |
|------|------|------|
| 核心组件 | `GroupChat` + Manager + UserProxy | `RoundRobinGroupChat` / `SelectorGroupChat` |
| 启动方式 | `UserProxy.initiate_chat` | 直接对群 `run(task=...)` |
| 发言顺序 | Manager 的 `speaker_selection_method` | 由群类型决定 |
| 散会条件 | 员工个人 `is_termination_msg` | 独立 `termination_condition` |

**哲学**：群聊从"需要 Manager 驱动的状态"变成"可独立执行的 Team 实体"。

## 🔟 群聊恢复：一键保存整队记忆

**v0.2 痛点**：必须手动抠出 `messages` 存盘、重启后再塞回，且只保存表面聊天记录（隐藏状态全丢）。

**v0.4 新办法**：

```python
# 第一天
group_chat = create_team(model_client)
stream = group_chat.run_stream(task="写个机器人故事。")
await Console(stream)

state = await group_chat.save_state()
with open("group_chat_state.json", "w") as f:
    json.dump(state, f)

# 第二天
group_chat = create_team(model_client)
with open("group_chat_state.json", "r") as f:
    state = json.load(f)
await group_chat.load_state(state)

stream = group_chat.run_stream(task="把故事翻译成中文。")
await Console(stream)
```

**保存的不仅是聊天记录**，而是**整个团队所有成员的"完整大脑快照"**（含内部状态、工具调用、思考过程）。

**应用**：长耗时任务（断点续跑）、多轮异步交互（用户离开后存档、回来读档）。

**提示**：若**不重启程序**（不重新 `create_team`），直接对原 `group_chat` 再调一次 `run_stream(新任务)` 即可，无需存档读档。

## 1️⃣1️⃣ 群聊内工具：绑给 Agent 本人

**v0.2 老办法（网管传话筒）**：

- 工具绑给专门的 `UserProxyAgent`。
- 群里的"写手 AI"需查天气时，**在群里发出 JSON 工具调用指令**。
- `Manager` 路由给 `UserProxy` 执行。
- 😵 **灾难 1**：Manager 经常路由失败，把工具调用发给了别的 AI。
- 😵 **灾难 2**：不支持 Function Calling 的老模型看到 JSON 指令会**胡言乱语**。

**v0.4 新办法（自带笔记本）**：

```python
writer = AssistantAgent(
    name="writer",
    model_client=model_client,
    tools=[get_weather],  # 谁用工具绑给谁
    reflect_on_tool_use=True,
)
```

- AI **不会在群里乱喊机器指令**。
- **内部**调用 `get_weather` → 拿到结果 → 包装成**正常文本消息**发到群里。
- 群里其他成员（即使不支持 Function Calling）看到的也是**正常文本**。

**对比**：

| 维度 | v0.2（网管传话筒） | v0.4（自带笔记本） |
|------|------------------|------------------|
| 工具绑定对象 | `UserProxyAgent` | 需要工具的 `AssistantAgent` |
| 谁执行 | UserProxy 代执行 | Agent 内部执行 |
| Manager 作用 | 必须路由工具调用 | 完全不管 |
| 群聊消息格式 | 含 JSON 工具调用指令 | 只有 AI 润色后的纯文本 |
| 对老模型兼容性 | ❌ 灾难 | ✅ 完美 |

**哲学**：从"系统级代理"（UserProxy 集中管控）走向"Agent 自治"（个人技能自己处理）。

## 1️⃣2️⃣ 自定义发言顺序：硬规则 + LLM 智能

**场景**：3 人团队——项目经理（`PlanningAgent`）+ 搜索员（`WebSearchAgent`）+ 数据分析师（`DataAnalystAgent`）。任务不能"自由散漫"抢麦，需星型汇报结构。

**v0.4 核心组件**：`SelectorGroupChat(selector_func=...)`

- 函数返回**具体名字** → 框架**强制执行**（硬规则）。
- 函数返回 **`None`** → 框架让 **LLM 智能判断**（软规则）。

**示例**（星型汇报：员工干完活必须先汇报给经理）：

```python
def selector_func(messages) -> str | None:
    # 铁律 1：只要上一条不是经理发的（说明员工刚干完活）
    if messages[-1].source != planning_agent.name:
        return planning_agent.name  # 强制交给经理

    # 铁律 2：经理刚派完活
    return None  # 让 LLM 智能选下一个员工
```

**运转流程**：

1. 老板下达任务 → 函数强制经理先发言。
2. 经理派活："@搜索员，去查数据"（上一条是经理 → `None`）。
3. LLM 智能判断 → 搜索员上场。
4. 搜索员发数据（上一条不是经理 → 强制经理）。
5. 经理汇总："@分析师，算百分比"（`None`）。
6. LLM → 分析师上场。
7. 分析师发结果（强制经理）。
8. 经理总结 → "TERMINATE" → 下班。

**哲学**："用代码写死底线（防呆），用大模型处理细节（智能）"。

## 1️⃣3️⃣ 嵌套群聊：包工头模式

**核心概念**：在"部门经理（Nested Agent）"的内部藏一个"内部小团队（Nested Team）"。大群其他成员**看不到**内部团队在干嘛（信息隔离），只看到经理交付的最终结论。

**v0.2 老办法（填表申请外包）**：调用 `register_nested_chats`，传一堆字典配置。死板，难做中间逻辑判断。

**v0.4 新办法（开外包公司）**：把"嵌套"做成**自定义 Agent 的内在技能**，在 `on_messages` 里手动调内部 Team 的 `run`。

**示例**（数数外包团队）：

```python
class CountingAgent(BaseChatAgent):
    """底层女工：收到数字 +1 喊出来"""
    async def on_messages(self, messages, ...):
        last_number = int(messages[-1].content)
        return Response(chat_message=TextMessage(content=str(last_number + 1)))


# 内部小团队（封闭车间）
counting_team = RoundRobinGroupChat([CountingAgent("女工1"), CountingAgent("女工2")], max_turns=5)


class NestedCountingAgent(BaseChatAgent):
    """包工头：外部消息转手给内部车间，最后只汇报结论"""
    def __init__(self, name, counting_team):
        self._counting_team = counting_team

    async def on_messages(self, messages, ...):
        result = await self._counting_team.run(task=messages)
        return Response(
            chat_message=result.messages[-1],   # 拿车间最后一句汇报
            inner_messages=...,                 # 内部流水可查账
        )
```

**效果**：外部大老板给包工头发"1" → 内部女工轮流喊"2, 3, 4, 5, 6"（5 轮）→ 包工头对外只汇报"6"。

**价值**：

- **信息隔离**：避免大群上下文爆炸（突破 Token 限制）。
- **黑盒封装**：大群不需要知道内部结构。
- **Society of Mind**：终极愿景——用 AI 管 AI，用团队模拟大脑，可嵌套无数层。

## 1️⃣4️⃣ 长上下文：智能滑动窗口

**痛点**：LLM 有 Token 限制，长对话会**直接死机**或**疯狂烧钱**。

**v0.2 老办法（外挂碎纸机）**：Agent 创建后手动挂 `transforms` 插件。社区吐槽：基础生存技能凭什么要打补丁？

**v0.4 新办法（内置智能记事本）**：

- 核心组件：`ChatCompletionContext`（上下文管理器）——Agent 的**出厂标配**。
- 高级概念：**虚拟视图（Virtual View）**——底层完整保存全部历史，调用 LLM 时只喂虚拟视图。

**示例**（滑动窗口）：

```python
assistant = AssistantAgent(
    name="assistant",
    system_message="You are a helpful assistant.",
    model_client=model_client,
    model_context=BufferedChatCompletionContext(buffer_size=10),  # 只保留最近 10 条
)
```

**运转逻辑**：

- 用户说第 11 句话时，框架自动把**第 1 句话（最老）踢出记事本**。
- 大模型视角：永远只看到最近 10 条。
- 配合 `while True: input(...)` 死循环聊天室：AI 永不爆脑。

**优势**：

- 原生基建（初始化传参即可）。
- 虚拟视图解耦：底层数据完整保留，可写自定义 Context（如 `SummaryContext` 压缩、`CodeOnlyContext` 过滤）。
- 防止"注意力稀释"（Lost in the middle）——限制 10~20 条反而提升回答质量。

## 1️⃣5️⃣ 工程化：监工 / 急停 / 日志 / 安全沙箱

### 👁️ 可观察性

- **盯单个员工**：`on_messages_stream()` → 给员工戴"脑电波头盔"，内部思考/工具调用实时流出。
- **盯整个团队**：`run_stream()` → 会议室实时监控摄像头，前端 UI 可直接接流做"打字机效果"。

### 🛑 控制力

- **急停按钮**：`CancellationToken` → 老板手里的遥控器，触发即异步中断输出流，让 AI 闭嘴。
- **自动下班**：`Termination conditions`（`TextMentionTermination`、`MaxMessageTermination`）→ 自动停。

### 📝 日志

- **v0.2 老办法**：自创"特殊日志模块"，难以接入 ELK / Datadog。
- **v0.4 新办法**：全面拥抱 Python 原生 `logging` 模块，专业后端友好。

### 💻 代码执行器升级

- **全面异步** + **一键 Kill 进程**：用 `CancellationToken` 杀掉 AI 的死循环代码。
- **云端防爆沙箱**（`ACADynamicSessionsCodeExecutor`）：
    - AI 代码发到 Azure 容器跑，碰不到本地文件。
    - 跑完即焚，每次全新隔离容器。
    - 彻底杜绝 `rm -rf /` 删库跑路。

## 💡 总结：v0.4 的 5 大设计哲学

1. **分层解耦**——Core（底层 Actor 框架）+ AgentChat（高层任务驱动）。
2. **OOP 取代回调**——继承 `BaseChatAgent` 重写方法，告别 `register_reply` 黑盒。
3. **实体化团队**——群聊是活的 Team，规则独立，对 Team 直接 `run` / `save_state`。
4. **Agent 自治**——工具和状态都绑给 Agent 本人，去掉 `UserProxy` 与 Manager 路由。
5. **原生工程化**——流式输出、CancellationToken、标准 logging、云端沙箱，企业级开箱即用。
